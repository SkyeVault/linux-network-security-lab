# What I Learned About Cyber Attacks by Running a Web Server

I've been running my own web server for a while now - hosting websites, a mail stack, an IRC network, a mailing list, and a handful of other services. I did it to learn. I did not expect how fast the internet would try to teach me back.

Here's what actually happened, and what I took away from it.

---

## The .git Incident

A few weeks ago I was reviewing traffic on my server and noticed something odd in tcpdump - 20+ simultaneous HTTPS connections from a single IP, all with data queued but not sending. I dug in.

Turned out it was a coordinated automated scanner running across four Google Cloud VMs, cycling through hundreds of randomized User-Agent strings (BlackBerry 9800, Nokia 5700, OS/2 Warp - the works) to avoid detection. It was probing every common webroot path looking for exposed `.git/config` files.

`/backend/.git/config` → 200 OK  
`/api/.git/config` → 200 OK  
`/.git/config` → 200 OK  

It found one. My `/var/www/.git/config` was publicly accessible because at some point I had cloned a repo directly into my webroot instead of outside it. Classic mistake.

The file only contained a GitHub URL - no credentials, no keys. Damage was minimal. But it was a real wake-up call.

**What I did:** Blocked all four IPs, deleted the `.git` directory, added an nginx rule to deny all `/.git/` access across every site, verified the fix, wrote an incident report.

**What I learned:** Scanners like this run constantly against every public IP. They are fast, they are automated, and they are looking for exactly the kind of small mistake anyone can make when they're learning.

---

## The 600 Gbps DDoS

A few months before that, my server went offline. Not because anything broke - because the upstream provider got hit with a 600 Gbps ransom DDoS that saturated the entire network.

My box was fine. CPU never went above 27%. RAM was comfortable. The server just couldn't be reached because the pipe was full before traffic even arrived.

Before I knew it was a provider-level event, I watched the attack evolve in real time:

- UDP flood → DNS water torture → TCP SYN flood → HTTP flood → back to DNS water torture with randomized subdomain casing to bypass caching

Someone was at a keyboard adapting the attack in response to mitigations. That was the detail that stuck with me most. This wasn't a script. It was a person.

**What I did:** Moved DNS to Cloudflare (removes the authoritative DNS target from my infrastructure entirely), hardened the mail stack, fixed fail2ban which had been silently not working, cleaned up orphaned services and Docker containers, and documented everything.

**What I learned:** At a certain scale, local mitigations don't matter. The pipe is full before your firewall sees a single packet. The real defense is your provider's upstream scrubbing capacity - something worth researching before you need it.

## The Smaller Stuff That Added Up

Along the way I found things I hadn't realized were problems:

- **fail2ban was running but silently doing nothing** - it was configured to watch the systemd journal but my mail logs went to syslog. Zero matches for months.
- **xrdp (remote desktop) was listening publicly** - a service I'd set up once and forgotten about, sitting on port 3389, one of the most commonly scanned ports on the internet.
- **Raw iptables rules I added during the DDoS weren't actually firing** - they were appended below UFW's chains and never evaluated. I thought I was blocking IPs. I wasn't.
- **opendmarc was connected to Postfix via Unix socket** - which doesn't work when Postfix runs in a chroot jail. Mail was being accepted without DMARC checks, silently, because `milter_default_action = accept`.

None of these were dramatic. All of them were invisible until I looked.

---

## What Running a Server Actually Teaches You

You can read about security concepts for years. Running a server puts you in the situation where the concepts become real problems with real consequences.

You learn what tcpdump output actually looks like when someone is scanning you. You learn what it feels like to block an IP and then watch three more show up. You learn that "security" is less about any single tool and more about knowing your own infrastructure well enough to notice when something is wrong.

The internet is not passive. The moment a server is reachable, it is being probed. Not personally - just automatically, constantly, by bots and scanners and scripts running against every IP in every subnet. The question isn't whether you'll be targeted. It's whether you'll notice.

Running your own server is one of the best ways I've found to actually understand that.

---

*If you're learning self-hosting, security, or Linux administration - I'd encourage you to run something real. Break it. Fix it. Document what happened. That's the whole curriculum.*
